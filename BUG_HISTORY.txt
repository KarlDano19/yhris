FIX EXPLANATION:
Three state variables and one disabled-logic update in `Checklist.tsx`:

**1. New `isRedoMode` state**

  const [isRedoMode, setIsRedoMode] = useState(false);

Tracks whether the user has clicked Redo on a saved approval. Reset in:
- `resetFormData()` — called on modal close
- The `currentStatus !== 'passed'` useEffect — so navigating away from Passed clears it

**2. State 3 render gate changed**

  // Before
  if (approval?.signature) { ... read-only block ... }

  // After
  if (approval?.signature && !isRedoMode) { ... read-only block with Redo button ... }

When `isRedoMode=true` the check is false, so execution falls through to the State 2
block where `showInlineForm=true` causes the editable form to render.

**3. Redo button added to the State 3 header**

The read-only block header was refactored from a single flex row into a
`justify-between` wrapper so the Redo button sits flush right:

  <div className='flex items-center justify-between'>

      <CheckBadgeIcon className='w-4 h-4' />
      Stage Approval
      {approval.is_skipped && <span ...>SKIPPED</span>}
    </div>

      <button
        type='button'
        onClick={() => {

          setIsSkipMode(false);
          setShowInlineForm(true);

  // Before

Without this change, `serverHasApproval=true` would keep the button enabled during
redo mode even though the user hasn't submitted a new signature yet. The added
`|| isRedoMode` term forces the button disabled until `pendingApproval` is set by
clicking "Submit Approval" in the inline form.

BUG ID: HRIS-021
DATE FILED: 2026-04-17
DATE FIXED: 2026-04-17
CLIENT/REPORTER: Internal - HR

ISSUE DESCRIPTION:
In the Screen Applicants Checklist modal, the "Save Changes" button had incorrect
disabled logic with two related problems:

1. The button was always disabled whenever the applicant status was "passed",
   regardless of whether a stage approval had been set. This blocked saving even
   when the user legitimately needed to save (e.g., after completing the inline
   approval form).

2. After fixing problem 1, a second gap was identified: when status is "passed"
   but the user has not yet chosen either "Skip Stage" or "Approve Stage" (State 1
   — showing the Skip/Approve buttons), the Save Changes button remained enabled.
   The form submit handler already had a silent early-return guard in this state,
   but the button itself gave no visual feedback that it was blocked.

Combined effect: the button was either always disabled (original bug) or not
visually disabled when it should be (gap after first fix), giving confusing UX
in both directions.

DESIRED OUTCOME:
The "Save Changes" button disabled state must follow these rules when status is "passed":

| State                                        | Disabled? | Tooltip                                         |
|----------------------------------------------|-----------|--------------------------------------------------|
| No action taken (Skip/Approve buttons shown) | Yes       | "Please complete stage approval before saving." |
| Inline form open, not yet submitted          | Yes       | "Please complete stage approval before saving." |
| "Submit Approval" clicked (pendingApproval)  | No        | —                                               |
| Approval already saved to server (signature) | No        | —                                               |

State 4 is ENABLED because the user may still need to save other fields (e.g., stage
notes). The server-saved approval is already persisted; blocking Save in that state
serves no purpose and strands users after a page refresh.

For all other statuses (ongoing, withdrawn, rejected, pooling), Save Changes is
always enabled (subject to loading state only).

FIX EXPLANATION:
The footer Save Changes button logic in Checklist.tsx was rewritten in two iterations.

**Iteration 1 (first partial fix):**
The original condition `isPassed` (always disabled when passed) was replaced with
`isPassed && hasApproval` where `hasApproval = !!getCurrentStageApproval()?.signature`.
This allowed saving when status was "passed" and a pending approval had been staged.
However it left the button enabled in State 1 (no action taken yet).

**Iteration 2 (partial fix):**
The condition was rewritten to:

  const serverApprovalSignature = !!applicant?.stage_approvals
    ?.find((a) => a.job_stage === actionState.stageId)?.signature;
  const isSaveDisabled = isPassed && (!pendingApproval || serverApprovalSignature);

This correctly handled States 1–3 but incorrectly kept State 4 (server approval
already present) disabled, even after a page refresh — stranding the user with a
permanently blocked Save button despite the approval already being persisted.

**Iteration 3 (final fix):**
The root insight: `pendingApproval` is local component state (lost on refresh) while
`applicant.stage_approvals` is server data (persists). State 4 should be ENABLED
because the approval is already saved; there is nothing left to guard against.

Final condition:

  const serverHasApproval = !!applicant?.stage_approvals
    ?.find((a) => a.job_stage === actionState.stageId)?.signature;
  // Disabled only when passed + no approval at all (neither local nor server)
  const isSaveDisabled = isPassed && !pendingApproval && !serverHasApproval;

Logic breakdown:
- State 1/2: `pendingApproval` is null AND `serverHasApproval` is false → disabled.
- State 3: `pendingApproval` is set → `!pendingApproval` is false → enabled.
- State 4: `serverHasApproval` is true → `!serverHasApproval` is false → enabled.
  (Also covers page-refresh scenario where pendingApproval is null but server has data.)

Single tooltip: "Please complete stage approval before saving." — shown only in
States 1/2 when the button is disabled. No separate "Cannot modify" message needed.

DATE FIXED: 2026-04-17

ISSUE DESCRIPTION:
Two visual bugs on the Talent Search page affecting applicant profile cards:

1. PlaceholderAvatar (SVG initials avatar) rendered oversized — filling its entire

   `height={size}` Next.js props (which define intrinsic dimensions / aspect ratio)
   but no CSS size constraint. Tall portrait images were not clamped and overflowed
   their intended 120×120 area.

DESIRED OUTCOME:
- PlaceholderAvatar renders at exactly the pixel dimensions passed via `width` and
  `height` props regardless of container size.
- Real photos are cropped to a square (120×120) via `object-cover` so all cards
  maintain the same height regardless of the photo's original aspect ratio.

DATE FIXED: 2026-04-06
CLIENT/REPORTER: Client - GALAM

ISSUE DESCRIPTION:
Clicking "Confirm" on the "Save changes?" modal in the Employment Details tab of
Employee 201 Records crashed with the error:

  "saveEmployment is not a function"

Position and Department (and all other Employment Details fields) could not be
saved. The same latent bug existed for the Personal Information tab
(savePersonal was also undefined) but had not yet been reported.

ROOT CAUSE:
`useEmploymentDetailsPatch` and `usePersonalDetailsPatch` both returned the raw
`useMutation` result from React Query:

  return useMutation<...>(...);

`useMutation` returns { mutate, mutateAsync, isLoading, ... } — there is no
`save` property. However, `Content.tsx` destructured both hooks expecting a
`save` function:

  const { save: savePersonal } = usePersonalDetailsPatch(params?.id);
  const { save: saveEmployment } = useEmploymentDetailsPatch(params?.id);

Because `save` was `undefined` on both, calling `saveEmployment(payload)` inside
`saveEmploymentTab()` threw "saveEmployment is not a function". The two files
fell out of sync — whoever wrote Content.tsx expected the hooks to wrap
`mutateAsync` in a `save` helper, but the hooks were never updated to do so.

DESIRED OUTCOME:
- Saving Employment Details (position, department, date hired, etc.) works
  without errors.
- Saving Personal Information continues to work correctly.

FIX EXPLANATION:
Both hooks were updated to wrap `useMutation` in a local variable and expose a
`save` async function that calls `mutation.mutateAsync(payload)` and returns
`{ ok: true }` on success or `{ ok: false, error }` on failure — matching the
contract expected by `savePersonalTab` and `saveEmploymentTab` in Content.tsx.

useEmploymentDetailsPatch.ts (same pattern applied to usePersonalDetailsPatch.ts):

  const mutation = useMutation<EmploymentPatch, Error, EmploymentPatch>(...);

  const save = async (payload): Promise<{ ok: boolean; error?: Error }> => {
    try {
      await mutation.mutateAsync(payload);
      return { ok: true };
    } catch (error) {
      return { ok: false, error: error as Error };
    }
  };

  return { ...mutation, save };
