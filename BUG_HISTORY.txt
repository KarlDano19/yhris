FIX EXPLANATION:
The bug was in the `processApplicants` function (Content.tsx:230-254) which evaluates
applicant screening answers against the job posting's ideal answers. The code called
`.toLowerCase()` directly on values that could be null:

1. Multiple Choice comparison (line 237): `app.toLowerCase() === ideal.toLowerCase()`
   - Both `app` (applicant's answer) and `ideal` (ideal answer) could be null

2. Yes/No comparison (line 244): `applicantAnswer.toLowerCase() === idealAnswer.toLowerCase()`
   - Both values could be null if screening data was incomplete

The fix added null checks before calling `.toLowerCase()`:
- For Multiple Choice: Added guard `if (app == null || ideal == null) return false;`
  and wrapped values with `String()` for type safety
- For Yes/No: Added conditional check that sets `isMatch = false` when either value
  is null, otherwise performs the comparison with `String()` wrapper

This ensures applicants with null/missing screening answers are treated as "not fit"
for those specific questions (conservative approach) rather than crashing the page.
The root cause was likely incomplete screening question data in the database for
job posting #276.
